# Darkness 운영 및 유지보수 가이드

이 문서는 Darkness 프로젝트를 실행·분석·수정·검증할 때 따라야 할 기준을 정리한다. 각 폴더의 짧은 역할 설명은 해당 폴더의 `README.md`를 참고하고, 실제 변경 절차와 시스템 간 연결 관계는 이 문서를 기준으로 삼는다.

## 1. 기술 환경

- 프로젝트 형식: 구형 MSBuild 형식의 C# 콘솔 애플리케이션
- 대상 프레임워크: .NET Framework 4.7.2
- 출력 형식: Windows 콘솔 실행 파일
- 기본 해상도: 80칸 × 30줄
- 실제 메시지 영역: 테두리를 제외한 78칸 × 13줄
- 소스 네임스페이스: `Darkness`

새 C# 파일은 SDK 스타일 프로젝트처럼 자동 포함되지 않는다. 파일을 추가했다면 반드시 `Darkness.csproj`의 `<Compile Include="..." />` 항목에도 등록한다.

## 2. 빌드와 실행

### 빌드

```powershell
dotnet build Darkness.csproj --no-restore --configuration Release
```

Visual Studio의 MSBuild를 사용할 수 있다면 다음 명령도 가능하다.

```powershell
msbuild Darkness.csproj /t:Build /p:Configuration=Release
```

빌드 결과는 `bin/Release/Darkness.exe`에 생성된다. Debug 실행 파일이 이미 실행 중이면 Windows 파일 잠금 때문에 Debug 빌드가 실패할 수 있다. 실행 중인 게임을 종료하거나 Release 구성으로 검증한다.

### 실행 전 조건

- 콘솔 창이 80×30보다 작으면 게임이 시작되지 않는다.
- `Game.Config()`가 UTF-8 출력을 설정하므로 콘솔 인코딩을 임의로 제거하지 않는다.
- 소스가 정상인데 PowerShell에서 한글이 깨져 보일 수 있다. 터미널 표시만 보고 파일 인코딩을 일괄 변환하지 않는다.

### 빌드 산출물 관리

`bin`, `obj`, `.vs`는 생성물 또는 IDE 상태다. 소스 변경으로 취급하지 않으며 코드 리뷰와 문서 갱신 대상에서도 제외한다.

## 3. 실행 구조

전체 실행 흐름은 다음과 같다.

1. `Program.Main()`이 `Game`을 생성한다.
2. `Game`이 콘솔을 초기화하고 `GameManager`를 실행한다.
3. `GameManager`가 인트로, 플레이, 게임 오버, 종료 상태를 전환한다.
4. `PlayingState`가 영웅과 시작 아이템을 만들고 `Exploration`을 시작한다.
5. `Exploration`이 화면 선택을 `PlayerTurnCommand`로 변환한다.
6. `TurnManager`가 플레이어 행동, 몬스터 행동, 방 행동, 상태이상과 턴 종료를 순서대로 처리한다.

상태 전환이나 새 게임 초기값을 바꿀 때는 `GameManager.cs`를 먼저 확인한다. 한 번의 플레이 내부 흐름을 바꿀 때는 `Phase/Exploration.cs`와 `Turn/TurnManager.cs`를 확인한다.

## 4. 턴 처리 순서

일반 행동의 핵심 순서는 다음과 같다.

1. 행동이 턴을 소비하는지 확인한다.
2. 속박이 있다면 원래 행동을 강제 대기로 교체하고 속박 1스택을 소비한다.
3. 대상 슬롯과 스킬 대상을 검증한다.
4. 방어 자세 등 플레이어의 선행 효과를 적용한다.
5. 플레이어 행동을 공격·효과·투척·슬롯 변화 요청으로 변환하고 실행한다.
6. 현재 몬스터 목록을 스냅샷으로 잡는다.
7. 각 몬스터가 행동을 결정한다.
8. 몬스터 행동을 실행한다.
9. 방 전용 턴 행동을 실행한다.
10. 방 효과와 턴 종료 상태이상을 처리한다.
11. 쓰러진 몬스터와 파괴된 콘텐츠를 정리한다.
12. 턴 번호를 증가시키고 입력 상태로 돌아간다.

이 순서를 바꾸면 첫 공격에 죽은 몬스터가 행동하거나, 문이 너무 일찍 발견되거나, 상태이상 지속시간이 한 턴 어긋날 수 있다. `TurnManager` 수정 후에는 반드시 공격, 투척, 탐색, 방어, 스킬, 이동을 각각 시험한다.

이동은 일반 행동과 별도 경로를 사용한다. 이동 중에는 `hasty` 효과가 적용되고, 이동 차단기가 먼저 검사된다. `MoveInterceptionResult.ConsumesTurn`이 참인 차단은 이동에 실패해도 몬스터와 방이 행동하고 턴이 증가한다.

## 5. 데이터와 실행 코드의 구분

`Data`에는 이름, 설명, 수치, ID와 구성만 둔다. 실제 행동은 도메인 또는 행동 클래스에 둔다.

- 영웅 수치: `Data/Heroes`
- 아이템 수치와 결속 스킬: `Data/Items`
- 몬스터 수치: `Data/Monsters`
- 방 연결과 슬롯 구성: `Data/Rooms`
- 스킬 비용과 효과: `Data/Skills`
- 상태이상 정의: `Data/Effects`
- 함정 수치: `Data/Traps`
- 개별 상태이상 동작: `Effect/Effects`
- 몬스터 판단: `Monster/Behavior`

ID 문자열은 데이터와 팩터리를 연결하는 Key이다. 오타가 컴파일 오류가 아니라 실행 중 `Unknown ... instance` 예외로 나타날 수 있으므로 ID 변경 시 전체 검색을 수행한다.

## 6. 아이템 추가 및 변경

새 아이템을 추가할 때의 순서다.

1. `Data/Items/ItemData.cs`에 고유 ID와 `ItemType`을 추가한다.
2. 분류, 최대 중첩, 무게, 공격·방어, 내구도, 사용 가능 여부를 지정한다.
3. 사용 기능이 기존 기능이면 `ItemFunction`을 연결한다.
4. 새로운 사용 기능이면 `Phase/Exploration.UseItem()` 또는 전용 처리기를 확장한다.
5. 투척 효과가 있으면 `ThrowEffects`와 `ItemThrowResolver`의 지형 소실 규칙을 확인한다.
6. 획득처를 `RoomSlotContentFactory`의 상자·더미·몬스터 인벤토리에 연결한다.
7. 설명은 콘솔 표시 폭 70칸 이하로 작성한다.

아이템을 버리거나 투척할 때는 반드시 `Inventory.Discard()`를 통과해야 한다. 장비를 직접 null로 만들거나 스택 수를 직접 수정하면 임시 인벤토리 용량이나 파손 처리가 어긋날 수 있다.

돌멩이는 14번 방 최초 진입 때 3개가 지급된다. 인벤토리가 가득 찼다면 임시 슬롯 하나가 생기고 마지막 돌멩이가 인벤토리에서 사라질 때 제거된다. 이 규칙을 수정할 때는 가득 찬 인벤토리, 부분 소모, 마지막 투척을 모두 검증한다.

## 7. 장비와 내구도

장비 장착과 해제는 `Hero.Equip()`과 `Hero.Unequip()`을 사용한다. 결속 스킬은 장비를 처음 장착할 때 영구 학습되며, 같은 스킬을 다시 배우지 않는다.

내구도 처리는 `DurabilityResolver`가 담당한다.

- 무기로 명중시키면 공격에 사용한 무기 내구도가 감소한다.
- 일반 피격 시 방어구 내구도가 감소한다.
- 방어 자세로 공격을 막으면 무기 내구도가 감소한다.
- 내구도가 0이 되면 장비 또는 인벤토리에서 제거되고 파손 메시지가 출력된다.
- 숫돌은 장착한 무기가 파손되기 전에 사용해야 한다.

내구도 규칙을 바꿀 때는 명중, 회피, 방어 자세, 투척 무기, 스킬 공격을 따로 검증한다.

## 8. 스킬 추가 및 변경

1. `Data/Skills/SkillData.cs`에 고유 ID와 `SkillType`을 추가한다.
2. 비용 종류와 양, 대상 방식, 공격 방식, 효과 목록을 지정한다.
3. 새 효과가 필요하면 `EffectApplication`과 `EffectResolver`가 해당 연산을 지원하는지 확인한다.
4. 새 상태이상이라면 `EffectData`, `ActiveEffectFactory`, `Effect/Effects`를 함께 갱신한다.
5. 장비로 배우는 스킬은 `ItemType.BoundSkillId`에 연결한다.
6. 스킬 설명은 70칸 이하인지 확인한다.
7. 비용 부족, 잘못된 대상, 장비 없음, 정상 사용을 각각 시험한다.

스킬이 붙은 장비를 처음 장착하면 `SkillMessages.Learned()`가 출력된다. 재장착 시에는 중복 메시지가 나오지 않아야 한다.

현재 플레이어의 집중력 회복 수단은 다음과 같다.

- 4번 방 휴식공간 진입: 최대치까지 회복
- 마력회상: 마석 1개를 소모해 집중력 3 회복
- 전투 중 방어 자세: 집중력 1 회복

사슬결박은 집중 3을 소모하고 속박 2스택을 부여한다. 속박은 대상의 다음 행동을 스택당 한 번씩 강제 대기로 바꾼다.

## 9. 상태이상 추가 및 변경

새 상태이상을 만들 때는 다음 세 부분을 모두 확인한다.

1. `Data/Effects/EffectData.cs`: 이름, 설명, 지속시간, 중첩 가능 여부와 최대치
2. `Effect/Effects`: 피해·명중·방어·턴 종료 등 실제 동작
3. `Effect/ActiveEffectFactory.cs`: ID와 구현 클래스 연결

효과 적용은 즉시 객체를 바꾸지 않고 `EffectRequest`, `EffectPlan`, `EffectResolver`를 통과시키는 것을 우선한다. 그래야 적용 실패, 대상 호환성, 중첩 메시지가 일관되게 처리된다.

방에 종속된 효과는 `IRoomEffectSource`와 출처 객체를 유지해야 한다. 맹독안개 중독은 함정이 같은 방에 있고 활성 상태일 때만 유지되며 턴 종료마다 방어 무시 피해 3을 준다.

젖음과 얼어붙음처럼 조합되는 효과는 적용 순서와 동일 대상 재사용 제한을 함께 시험한다.

## 10. 몬스터 추가 및 변경

1. `MonsterData`에 능력치 원형을 추가한다.
2. 일반 행동으로 충분하면 `DefaultMonsterBehavior`를 사용한다.
3. 고유 판단이 필요하면 `Monster/Behavior`에 행동 클래스를 만든다.
4. `RoomSlotContentFactory`에 방 인스턴스 ID와 생성 함수를 등록한다.
5. 전리품은 몬스터가 가진 `Inventory`에 넣는다.
6. 첫 상태가 무관심, 경계, 전투 중 무엇인지 명시한다.
7. 탐색, 대화, 공격, 투척, 대기, 방어, 이동에 대한 반응을 확인한다.

행동 메시지에 몬스터 이름을 직접 넣기보다 `NarrativeTokens.Actor`를 사용한다. 미공개 슬롯에서는 이름이 `???`로 가려지고 공개 후 실제 이름으로 치환된다.

몬스터가 슬롯 사이를 이동한다면 `ISlotMovementBehavior`를 구현하고 진입 가능한 콘텐츠와 떠난 자리에 남길 콘텐츠를 정의한다.

## 11. 방과 슬롯 추가 및 변경

1. `RoomData`에 `RoomType`을 추가한다.
2. 고유한 방 ID, 이름, 설명과 진입 문구를 작성한다.
3. 방향별 `RoomEdgeType`을 연결한다.
4. 슬롯은 UI 규칙상 반드시 5개로 유지한다.
5. 각 슬롯에 `RoomObjectType`, 인스턴스 ID, 출구 여부를 지정한다.
6. 콘텐츠 ID를 `RoomSlotContentFactory`에 등록한다.
7. 방 전용 턴 행동이 필요하면 `IRoomTurnBehavior`와 팩터리를 갱신한다.
8. 이동을 가로채야 하면 `IMoveInterceptor`와 팩터리를 갱신한다.

문은 `HasDoor`만으로 즉시 이동 가능해지지 않는다. 슬롯이 공개되어 있고 콘텐츠가 문 발견을 막지 않을 때 `TryDiscoverDoor()`가 성공해야 한다. 콘텐츠 제거·이동 후 문 발견 메시지가 나오는지 확인한다.

모든 슬롯 공개 방은 진입 메시지도 밝은 환경에 맞아야 한다. 반대로 미공개 방의 메시지가 슬롯의 정체를 지나치게 직접 밝히지 않는지 확인한다.

## 12. 주요 특수 방의 불변 규칙

### 4번 방

휴식공간이다. 진입할 때마다 생명력과 집중력을 최대치까지 회복한다.

### 11번 방

처음 이동하려 할 때 문붙이가 출현하고 이동을 취소한다. 이 최초 차단은 현재 턴을 소비하지 않는다. 마석을 부위에 던지면 소멸한다.

### 14번 방

진동벌레는 탐색 또는 이동 진동에 즉사급 공격으로 반응한다. 돌멩이는 2회의 행동 여유를 만들며 모래에 던지면 회수할 수 없다. 최초 진입 지급과 임시 인벤토리 슬롯을 함께 유지한다.

### 15번 방

이전 방으로 돌아갈 수 없다. 안전 턴이 지난 뒤 숨결사냥꾼에게 즉사하므로 방 행동 카운터를 변경할 때 모든 조사 경로를 다시 계산한다.

### 16번 방

출구는 3번째 슬롯, 즉 인덱스 2의 복부 위치에 있다. 거미줄을 없애 문을 발견해도 복부가 살아 있으면 이동할 수 없다. 차단된 이동은 턴을 소비하고 보스가 행동한다. 복부 처치 후에만 모든 부위가 정리되고 출구로 이동할 수 있다.

## 13. 슬롯 콘텐츠 추가

새 환경 대상은 보통 `ISlotContent`를 구현한다. 필요에 따라 다음 계약을 추가한다.

- `IDamageable`: 공격과 투척 피해를 받음
- `IEffectTarget`: 상태이상을 받음
- `IDestroyableSlotContent`: 파괴 후 별도 결과를 만듦
- `INonBlockingTerrain`: 콘텐츠가 있어도 문 발견을 막지 않음
- `ISlotAppearance`: 미공개 상태에서도 모래처럼 외형 이름을 표시함
- `IPlayerTargetability`: 플레이어 직접 공격 가능 여부를 제어함
- `IRoomEffectSource`: 방 전체 효과의 출처가 됨

새 `RoomObjectType`을 만들었다면 열거형, 데이터, `RoomSlotContentFactory.factories`, 실제 클래스와 프로젝트 파일을 모두 갱신한다.

## 14. 인벤토리와 전리품

아이템 이동은 `Inventory.Store`, `Discard`, `TransferTo`, `TransferAllTo`를 사용한다. 각 함수의 반환값은 저장하지 못한 수량 또는 실제 이동 수량이므로 무시하지 않는다.

검증해야 할 경우는 다음과 같다.

- 빈 인벤토리
- 슬롯은 가득 찼지만 같은 아이템 스택에 공간이 있는 경우
- 최대 중첩을 넘어 새 슬롯이 필요한 경우
- 일부만 옮길 수 있는 경우
- 장비 해제 시 돌아올 공간이 없는 경우
- 투척 실패로 아이템을 복구해야 하는 경우
- 임시 용량을 만든 아이템의 마지막 수량이 사라지는 경우

바닥 인벤토리는 슬롯마다 존재한다. 물과 모래에 던진 아이템은 바닥으로 가지 않고 소실된다.

## 15. 메시지와 콘솔 UI

사용자에게 보이는 공통 문구는 가능하면 `Narrative`에 둔다. 콘텐츠 고유 묘사는 해당 콘텐츠에 둘 수 있지만 같은 뜻의 문구가 반복되면 메시지 클래스로 옮긴다.

### 폭 제한

- 콘솔 전체 폭: 80칸
- 메시지 뷰 내부 폭: 78칸
- 선택지 설명 안전 기준: 70칸 이하
- 70칸 이하인 기존 설명은 불필요하게 수정하지 않는다.

진행 메시지는 `Utility.WrapText()`로 자동 줄바꿈된다. 선택지 설명은 자동 줄바꿈되지 않으므로 반드시 70칸 이하로 유지한다. 한글은 `Utility.GetDisplayWidth()`에서 일반적으로 2칸으로 계산된다.

긴 전투 문구는 잘리지는 않지만 줄마다 Enter 입력이 필요하다. 늑대 전술처럼 여러 정보를 한 문장에 합치지 말고 상태 변화와 현재 행동을 별도 메시지로 나누는 편이 좋다.

### 문맥 점검

- 미공개 콘텐츠 이름은 기본적으로 `???`여야 한다.
- `ISlotAppearance`가 있으면 화면과 행동 문구 모두 외형 이름을 사용해야 한다.
- 행동 시작 문구만 나오고 대상 반응이 비어 있지 않은지 확인한다.
- 공개된 보스 부위를 계속 `무언가`라고 부르지 않는다.
- 공격했는데 이동 반응만 설명하는 식의 행동·결과 불일치를 피한다.

## 16. 밸런스 변경 방법

한 수치를 바꿀 때는 관련 비용·보상·설명을 동시에 갱신한다.

예시:

- 스킬 비용 변경: `SkillData` 비용, 설명, 사용 가능 조건 테스트
- 상태이상 중첩 변경: 효과 적용 수, 최대 중첩, 강제 대기 횟수 테스트
- 몬스터 공격 변경: 방어구 유무, 방어 자세, 회피, 상태이상 조합 테스트
- 전리품 변경: 모든 경로의 획득 가능 수량과 인벤토리 용량 테스트
- 마석 변경: 획득량, 거래 지불, 스킬 비용, 문붙이 소모량을 한 경로별로 합산

평균적인 최적 행동만 시험하지 말고 초보자가 할 법한 탐색 반복, 잘못된 투척, 빈 슬롯 공격, 장비 미착용, 인벤토리 만석도 포함한다.

## 17. 코드 흐름 기반 회귀 테스트

실제 키 입력 없이 공개 API로 방과 영웅을 생성해 턴을 실행할 수 있다. 최소 회귀 범위는 다음과 같다.

### 전체 조합 스모크 테스트

각 방의 5개 슬롯에 대해 다음 행동이 예외 없이 끝나는지 확인한다.

- 대기
- 대화
- 탐색
- 공격
- 방어
- 보유 가능한 모든 아이템 투척
- 존재하는 모든 방향으로 이동

### 집중 검증 시나리오

- 14번 방에서 준비 없는 탐색은 진동벌레 공격으로 이어진다.
- 돌멩이 투척 후 문 발견과 이동까지 2회의 여유가 유지된다.
- 마지막 돌멩이 소진 후 임시 인벤토리 슬롯이 제거된다.
- 15번 방에서 제한시간 이후 즉사한다.
- 16번 방에서 거미줄 제거 후 조기 이동이 차단된다.
- 16번 방의 복부 외 부위를 제거해도 이동은 차단된다.
- 복부 처치 후 17번 방으로 이동할 수 있다.
- 전투 중 방어 자세는 집중력 1을 회복하지만 빈 방에서는 회복하지 않는다.
- 사슬결박은 집중 3을 쓰고 속박 2스택을 적용한다.
- 맹독안개는 턴 종료마다 피해 3을 준다.

### 메시지 검증

- 모든 선택 설명의 표시 폭이 70 이하인지 검사한다.
- 대화·탐색·공격 후 출력 메시지가 완전히 비는 조합이 없는지 검사한다.
- 동적 이름이 들어간 스킬 결과가 78칸을 넘으면 정상적으로 줄바꿈되는지 확인한다.
- 같은 장비를 재장착할 때 스킬 습득 메시지가 반복되지 않는지 확인한다.

## 18. 변경 완료 체크리스트

변경을 마치기 전에 다음 항목을 확인한다.

- [ ] 새 ID가 데이터와 팩터리에서 정확히 일치한다.
- [ ] 새 C# 파일이 `Darkness.csproj`에 포함됐다.
- [ ] 수치와 사용자 설명이 서로 일치한다.
- [ ] 선택 설명이 70칸 이하이다.
- [ ] 성공·실패·대상 없음·인벤토리 만석 메시지가 있다.
- [ ] 행동이 턴을 소비하는지 의도와 일치한다.
- [ ] 상태이상 지속시간과 중첩 수가 의도와 일치한다.
- [ ] 문 발견과 이동 차단 순서가 유지된다.
- [ ] 아이템이 복제되거나 소리 없이 사라지지 않는다.
- [ ] Debug 또는 Release 빌드가 경고와 오류 없이 성공한다.
- [ ] 관련 방의 정상 경로와 실수 경로를 모두 재현했다.
- [ ] 폴더 역할이 바뀌었다면 해당 폴더의 `README.md`도 갱신했다.

## 19. 현재 구조상 주의할 부분

- `Trade`는 미완성 초기 구조다. 실제 갑주령 거래는 `ArmorSpiritBehavior`가 처리한다.
- `Data/StatusEffects`는 비어 있으며 실제 효과 데이터는 `Data/Effects`에 있다.
- 자동화된 단위 테스트 프로젝트가 없으므로 빌드 성공만으로 게임 규칙이 검증되지는 않는다.
- `TurnManager`는 많은 시스템을 조율하는 큰 클래스다. 수정 범위를 작게 유지하고 처리 순서를 임의로 이동하지 않는다.
- 일부 방은 슬롯 인덱스에 의미가 있다. 슬롯 순서를 바꿀 때 행동 클래스의 인덱스도 함께 갱신한다.
- 콘솔 메시지의 조사 표기는 `을(를)`, `이(가)` 형식을 사용한다. 자연스러운 문장이 중요한 고유 메시지는 완성된 문장으로 별도 작성한다.

이 문서는 현재 구현을 기준으로 한다. 규칙이나 구조를 변경한 경우 코드와 같은 작업에서 이 문서도 함께 수정한다.
